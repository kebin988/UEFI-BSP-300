# Reset Vector 揭秘：为什么 x86 从 0xFFFFFFF0 开始执行？

## 写在前面

上一篇我们了解了 SEC 阶段的整体流程。你知道了 CPU 从 `0xFFFFFFF0` 取第一条指令。

但你有没有想过——**为什么偏偏是这个奇怪的地址？**

为什么不是 `0x00000000`（地址空间开头）？为什么不是某个好记的整数？为什么是 4GB 末尾倒数第 16 个字节这么别扭的位置？

今天我们就来扒一扒这个设计背后的历史原因和硬件逻辑。

---

## 一、从 8086 说起

1978 年，Intel 8086 诞生。它只有 **20 位地址线**，能寻址 **1MB** 内存（`0x00000` ~ `0xFFFFF`）。

8086 的 Reset Vector 在哪？

```
CS = 0xFFFF, IP = 0x0000
物理地址 = CS × 16 + IP = 0xFFFF0

也就是 1MB 空间的倒数第 16 个字节！
```

为什么在末尾？因为 **ROM 被映射在地址空间的顶部**：

```
8086 地址空间（1MB）：
0x00000 ┌─────────────────┐
        │                 │
        │   RAM (640KB)   │  ← 可读写内存
        │                 │
0xA0000 ├─────────────────┤
        │  Video Memory   │  ← 显存
0xC0000 ├─────────────────┤
        │  扩展 ROM       │  ← 扩展卡的 ROM
0xF0000 ├─────────────────┤
        │  System ROM     │  ← BIOS ROM（64KB）
0xFFFF0 │  Reset Vector!  │  ← CPU 第一条指令
0xFFFFF └─────────────────┘
```

**设计理由**：
- RAM 放底部 → 从 0 开始连续分配，方便程序使用
- ROM 放顶部 → CPU 上电直接执行 ROM 代码，无需先初始化 RAM

这个设计从 1978 年一直延续到今天，**40 多年没变**。

---

## 二、从 1MB 到 4GB：地址空间扩大了，习惯没变

| CPU | 年份 | 地址位宽 | 地址空间 | Reset Vector |
|-----|------|---------|---------|-------------|
| 8086 | 1978 | 20 bit | 1 MB | `0xFFFF0` |
| 80286 | 1982 | 24 bit | 16 MB | `0xFFFFF0` |
| 80386 | 1985 | 32 bit | 4 GB | `0xFFFFFFF0` |
| x86-64 | 2003 | 48/64 bit | 256 TB+ | `0xFFFFFFF0` (仍然!) |

规律很明显：**永远是地址空间末尾倒数第 16 字节**。

为什么不变？**向后兼容**。

- 所有 x86 CPU 上电后都进入 Real Mode（兼容 8086）
- Real Mode 只有 20 位地址线……才怪！实际上现代 CPU 上电时 CS.base 被特殊设置为 `0xFFFF0000`
- 这样 `CS.base + IP = 0xFFFF0000 + 0xFFF0 = 0xFFFFFFF0`

```
上电时的 CPU 状态（以 x86-64 为例）：

┌─────────── 表面上 ──────────────┐  ┌─────── 实际上 ─────────┐
│ Mode: Real Mode (16-bit)        │  │ CS.base = 0xFFFF0000   │
│ CS = 0xF000                     │  │ (不是 0xF000 × 16!)    │
│ IP = 0xFFF0                     │  │                         │
│                                 │  │ 实际取指地址:            │
│ "看起来"只有 1MB 寻址能力       │  │ 0xFFFF0000 + 0xFFF0     │
│                                 │  │ = 0xFFFFFFF0            │
└─────────────────────────────────┘  └─────────────────────────┘
```

> 💡 这是 x86 最经典的"兼容性 hack"之一——表面上是 Real Mode，暗地里访问 4GB 空间顶部。

## 三、为什么 Flash 在地址空间顶部？

现代 PC 中，SPI Flash 通过芯片组被映射到 **4GB 以下的顶部区域**：

```
4 GB 物理地址空间：

0x00000000 ┌──────────────────────────────┐
           │                              │
           │        DRAM (系统内存)        │
           │        (假设 8GB)             │
           │                              │
0x??000000 ├──────────────────────────────┤
           │     MMIO (设备寄存器映射)     │
           │     PCI Config Space          │
           │     LAPIC / IO-APIC           │
           ├──────────────────────────────┤
           │                              │
0xFF000000 │     SPI Flash 映射区          │ ← 16MB Flash
           │     (Firmware 代码在这里)     │
           │                              │
0xFFFFFFF0 │     Reset Vector              │ ← 第一条指令
0xFFFFFFFF └──────────────────────────────┘
```

**芯片组的工作**：

芯片组（PCH/南桥）收到 CPU 对 `0xFFFFFFF0` 的访问请求时：
1. 发现这个地址落在 "Flash 映射区" 范围内
2. 把访问转发给 SPI Flash 控制器
3. SPI Flash 返回数据给 CPU

CPU 完全不知道自己读的是 Flash——它以为就是普通内存。这就是**内存映射 I/O (MMIO)** 的威力。

## 四、16 字节能干什么？

Reset Vector 只有 16 字节的空间（从 `0xFFFFFFF0` 到 `0xFFFFFFFF`），因为再往后就没地址了。

16 字节能放什么指令？

```nasm
; 方案 1：远跳转（5 字节）
jmp far 0xF000:0xE000    ; 跳到 BIOS 入口

; 方案 2：短跳转（2 字节）
jmp short -14            ; 相对跳转到前面的代码

; 方案 3：近跳转（3 字节）  
jmp 0xFFFF0000           ; 跳到 Flash 中的 SEC 入口
```

实际上现代 UEFI 固件通常是一条**近跳转**或**远跳转**，立即跳到 Flash 中更大空间的 SEC 代码。

EDK2 中 Reset Vector 的实际内容（x64）：

```
Offset    Hex                   Instruction
────────  ────────────────────  ─────────────────
FFF0      90 90 ...             NOP (填充)
FFF0      E9 xx xx xx xx        JMP rel32 (跳转到 SEC 入口)
...
FFFA      xx xx xx xx           FIT pointer (Firmware Interface Table)
FFFE      xx xx                 填充/签名
```

## 五、为什么 ARM 的 Reset Vector 在 0x0？

ARM 选择了完全相反的设计：Reset Vector 在地址 **0x00000000**（或通过 RVBAR 寄存器可配置）。

```
ARM 的设计哲学：
- ROM/Flash 映射在地址空间底部
- Reset Vector 在 0x0
- 异常向量表也在低地址

x86 的设计哲学：
- ROM/Flash 映射在地址空间顶部  
- Reset Vector 在末尾
- RAM 从 0 开始
```

| | x86 | ARM |
|--|-----|-----|
| Reset Vector | 地址空间末尾 | 地址空间开头 |
| 设计年代 | 1978 (8086) | 1985 (ARM1) |
| 历史包袱 | 极重（40年兼容） | 较轻 |
| 可配置？ | 不可（硬编码） | 可（RVBAR） |
| Flash 位置 | 顶部 | 底部（或可配置） |

ARM 没有 x86 那种 40 年的兼容性包袱，所以设计更灵活：
- 你可以配置 Reset Vector 地址
- 不同的启动安全等级有不同的入口
- 不需要 "Real Mode → Protected Mode" 的切换把戏

## 六、RISC-V 的做法

RISC-V 更加自由——Reset Vector 完全由**硬件实现者决定**：

```
// RISC-V 规范只说了：
// "mhartid = 0 的 hart (核心) 从 RESET_VECTOR 开始执行"
// 但不规定 RESET_VECTOR 的具体值

常见实现：
- QEMU virt:  0x80000000
- SiFive U:   0x00001000  
- T-Head C910: 厂商自定义
```

## 七、一个有趣的问题：没有 DRAM 为什么能访问 Flash？

这是初学者常见的困惑：

> "内存还没初始化，CPU 怎么能读到 Flash 的数据？"

答案：**Flash 不需要初始化**。

```
DRAM (DDR4/DDR5)：
  - 需要配置内存控制器（几百个寄存器）
  - 需要内存训练（信号时序校准）
  - 不初始化 → 读到的全是垃圾

SPI Flash (NOR Flash)：
  - 上电即可读（Read-Only 天生可用）
  - 芯片组的 SPI 控制器有默认配置
  - 不需要任何软件初始化就能读
  - 写需要特殊命令，但读是免费的
```

所以启动顺序是有道理的：
```
1. CPU 上电 → 从 Flash 读代码（Flash 天生可读）
2. 执行 Flash 中的代码 → 初始化 DRAM
3. DRAM 可用后 → 把代码从 Flash 复制到 DRAM 执行（更快）
```

## 八、如果 Reset Vector 指向的地方没代码会怎样？

| 情况 | 结果 |
|------|------|
| Flash 为空（全 0xFF） | CPU 执行 `0xFF 0xFF...` = 无效指令 → Triple Fault → 重启 → 循环 |
| Flash 损坏 | 同上，或执行错误代码 → 死机 |
| Flash 被拔掉 | 总线返回全 1 或总线错误 → CPU 挂起 |
| Flash 内容正确 | 正常启动 |

> 这就是为什么刷 BIOS 失败可能"变砖"——Reset Vector 处的代码坏了，CPU 无法启动。

### 恢复手段

| 方法 | 原理 |
|------|------|
| 双 BIOS（Dual BIOS） | 第二块 Flash 有备份 |
| BIOS Recovery | 硬件强制从备份区启动 |
| SPI 编程器 | 外部设备直接写 Flash |
| Intel ME Recovery | ME 微引擎尝试恢复 |

## 九、总结

```
为什么 x86 从 0xFFFFFFF0 开始？

1978年：8086 设计者把 ROM 放在 1MB 顶部 → Reset Vector 在末尾
         ↓
    这个设计决策被固化了 40+ 年
         ↓
现代 CPU：即使有 48/64 位地址空间，上电仍从 0xFFFFFFF0 开始
         ↓
原因：向后兼容（x86 的命，不可更改）
```

| 概念 | 说明 |
|------|------|
| Reset Vector | CPU 上电执行的第一个地址 |
| x86 位置 | `0xFFFFFFF0`（4GB - 16 字节） |
| 历史原因 | 8086 的 ROM 在地址空间顶部 |
| 16 字节限制 | 只够放一条跳转指令 |
| Flash 映射 | 芯片组把 SPI Flash 映射到高地址 |
| 不需要初始化 | NOR Flash 上电即可读 |

> **一句话总结**：x86 从顶部启动是 1978 年的设计遗产，40 年来为了兼容性一字未改。ARM/RISC-V 没有这个包袱，想放哪就放哪。
