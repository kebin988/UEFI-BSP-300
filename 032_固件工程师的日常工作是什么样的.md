## 一、先破除一个误解

很多人以为固件工程师就是"写 BIOS 的"。

不完全对。固件工程师的工作内容，取决于你在产业链的哪个位置、做什么平台、服务什么产品。但有一些共同点——让我来还原一个固件/BSP 工程师的典型工作日。

## 二、一个固件工程师的典型一天

> 以下是一个在芯片方案公司做 ARM BSP 的工程师视角，x86 BIOS 工程师的日常在后面对比。

### 09:00 - 开机，先看 Bug 列表

打开 Jira（或飞书/禅道/Bugzilla），看看昨晚测试团队给你提了几个新 Bug：

```
[P1] 🔴 UFS 偶发性初始化失败，概率约 5%
[P2] 🟡 S3 恢复后 USB 设备不识别
[P3] 🟢 开机 Logo 显示位置偏移 30 像素
[P4] 🔵 ACPI 表中 CPU 核心数描述错误
```

P1 是紧急 Bug，今天必须定位。P3 和 P4 可以安排后面修。

### 09:30 - 接上串口，开始调试

抓起串口线，连上开发板。打开串口终端（PuTTY/minicom/MobaXterm），给开发板上电：

```
[SEC] SecStartup Entry
[PEI] Memory Init: DDR4 3200MHz, 8GB
[PEI] UFS Init... 
[PEI] UFS Init... TIMEOUT!           ← 问题在这里
[PEI] UFS Retry 1... OK
[DXE] DxeMain Entry
...
```

看到了——UFS 初始化第一次超时，重试后成功。5% 的概率复现，说明可能是**时序问题**。

### 10:00 - 翻代码、查寄存器

打开 EDK2 源码，定位到 UFS 初始化代码：

```c
// 简化示意
Status = UfsExecDeviceCommand (UfsHc, UFS_QUERY_NOP_OUT, ...);
if (EFI_ERROR (Status)) {
    // 重试
    gBS->Stall (1000);  // 等 1ms
    Status = UfsExecDeviceCommand (UfsHc, UFS_QUERY_NOP_OUT, ...);
}
```

怀疑是 `Stall` 时间不够。查高通的 UFS 控制器手册——果然，spec 说在特定电压条件下，UFS 设备上电后需要等待 **10ms** 才能接收 NOP OUT 命令。代码只等了 1ms。

### 11:00 - 改代码、编译、刷机

```c
// 修复：增加等待时间
gBS->Stall (10 * 1000);  // 等 10ms
```

保存，编译：

```bash
build -p QcomPkg/QcomPkg.dsc -a AARCH64 -t GCC5
```

编译完成，把新的固件镜像通过 fastboot/QFIL 刷到开发板：

```bash
fastboot flash xbl xbl.elf
fastboot reboot
```

### 11:30 - 跑 100 次开机测试

写个脚本让开发板循环重启 100 次，统计 UFS 初始化失败率：

```bash
for i in $(seq 1 100); do
    echo "=== Test $i ===" >> result.log
    # 串口抓 log、重启开发板
    ...
done
grep "UFS.*TIMEOUT" result.log | wc -l
```

结果：0 次失败。Bug 修复，提交代码。

### 14:00 - Code Review + 合并

把修复提交到 Gerrit（或 GitLab），写 Commit Message：

```
UFS: Increase NOP OUT wait time to 10ms after power-on

The UFS specification requires a minimum of 10ms delay after
power-on before sending NOP OUT commands. The previous 1ms 
delay caused intermittent initialization failures (~5%).

Bug: JIRA-12345
Tested: 100 reboot cycles, 0 failures
Change-Id: I1234567890abcdef
Signed-off-by: BSP Engineer <engineer@company.com>
```

等同事 Review 通过，合并到主分支。

### 15:00 - 客户打电话来了

```
客户："我们的产品用了你们的开发板，开机时间太长了（8 秒），能不能优化到 3 秒？"
```

拿出 boot timing 分析工具，看看时间花在哪：

```
[00.000] SEC Start
[00.050] PEI Start
[00.200] Memory Init Done
[02.500] UFS Init Done          ← 2.3 秒！
[02.800] DXE Start
[05.000] GOP Display Logo       ← 2.2 秒在 Display Init
[06.500] BDS Start
[07.800] OS Loader Handoff
[08.000] Windows Logo
```

问题定位：UFS 初始化 2.3 秒太长（可以用并行化优化），Display 初始化 2.2 秒也可以压缩。给客户写优化方案。

### 16:30 - 写文档

把今天的 Bug 修复写成内部 Wiki 文档——下次有人遇到类似问题可以直接查。这是 BSP 团队的知识库积累。

### 17:30 - 看邮件、升级 SDK

高通发了新版 BSP SDK，diff 了一下 release notes：

```
What's new in QCS6490 BSP v3.2:
- Fixed SMMU context bank leak under heavy DMA load
- Added Wi-Fi 6E firmware support  
- Updated TrustZone firmware to v2.7
- [Breaking] Changed UFS register base address mapping
```

最后一条是 Breaking Change，明天得评估影响范围。

## 三、x86 BIOS 工程师的一天（对比）

如果你是在 AMI/Insyde/Phoenix 做 x86 BIOS 的，日常画风会有些不同：

| 工作内容 | ARM BSP 工程师 | x86 BIOS 工程师 |
|---------|----------------|-----------------|
| **主要代码库** | 芯片原厂 BSP + EDK2 | AMI Aptio/Insyde H2O + EDK2 |
| **调试手段** | 串口 UART | 串口 + 80 Port Debug Card |
| **内存初始化** | DDR Training（原厂 blob） | Intel MRC/FSP-M |
| **典型 Bug** | DDR 时序、UFS/eMMC、SMMU、Display | MRC 训练失败、ACPI 表错误、S3 恢复 |
| **客户场景** | IoT 设备、手机、车机 | 笔记本、服务器、工控机 |
| **刷机方式** | fastboot/QFIL/jtag | SPI Flash 烧录器/BIOS 更新工具 |
| **跟谁打交道** | 高通 FAE、客户硬件团队 | Intel FAE、OEM/ODM 客户 |
| **ACPI 工作量** | 中（ARM 用 DT 较多） | 大（x86 重度依赖 ACPI） |

### x86 BIOS 工程师的独特挑战

1. **ACPI 表的噩梦**：x86 上 ACPI 表特别复杂，电源管理、热管理、设备枚举全靠 ACPI。一个 `_PSS`（处理器性能状态）写错了，Windows 蓝屏给你看
2. **Intel FSP 的坑**：FSP 是闭源 Blob，出了 Bug 得找 Intel FAE，自己没法调（类似高通的 XBL）
3. **兼容性地狱**：要兼容 N 年前的 Windows 7、各种奇葩外设、Legacy CSM 模式…ARM 平台一般只用管最新的系统

## 四、固件工程师的核心技能树

不管是 ARM 还是 x86，固件工程师需要这些能力：

```
固件工程师技能树
├── 必修课
│   ├── C 语言（精通指针、结构体、位操作）
│   ├── UEFI/EDK2 框架
│   ├── 硬件寄存器操作（MMIO、PIO）
│   ├── 串口调试（这是你最好的朋友）
│   ├── 看芯片手册的能力（比看代码更重要）
│   └── Git + Gerrit/GitLab 代码管理
│
├── 高频使用
│   ├── ACPI / Device Tree
│   ├── PCIe / USB / I2C / SPI 总线协议
│   ├── 内存初始化原理（DDR Training）
│   ├── 启动时间优化
│   └── 安全启动（Secure Boot / Measured Boot）
│
├── 加分项
│   ├── ARM TrustZone / Intel TXT
│   ├── 电源管理（PSCI / ACPI _CPC）
│   ├── 虚拟化（Hypervisor 启动）
│   ├── Linux 内核驱动（逆向参考）
│   └── Python / Shell 脚本（自动化测试）
│
└── 隐藏技能
    ├── 耐心（有些 Bug 要跑 1000 次才能复现）
    ├── 和硬件工程师沟通（"你确定这个管脚接对了？"）
    ├── 和 FAE 斗智斗勇（"这个 Errata 什么时候能修？"）
    └── 在 Jira 里写出让人看得懂的 Bug 报告
```

## 五、固件工程师的工作节奏

固件开发有明显的项目周期：

```
新项目立项
    │
    ├── 第1-2月：BSP Bring-up
    │   ├── 让芯片第一次跑起来（点灯大法好）
    │   ├── 内存初始化、串口输出、基本外设
    │   └── 这个阶段最有成就感："它活了！"
    │
    ├── 第3-4月：外设驱动集成
    │   ├── UFS/eMMC、Display、USB、PCIe、网络
    │   ├── 各种 Bug 涌现
    │   └── 每天都在"修一个 Bug 冒三个新 Bug"
    │
    ├── 第5-6月：性能优化 + 系统联调
    │   ├── 启动时间优化
    │   ├── 和 OS 团队联调（Windows/Linux/Android）
    │   └── 客户开始提各种定制需求
    │
    ├── 第7-8月：量产准备
    │   ├── 稳定性测试（跑 10000 次重启）
    │   ├── 安全功能验证（Secure Boot、fTPM）
    │   └── 生产线烧录工具和流程
    │
    └── 后续：维护 + 新项目
        ├── 客户反馈的零星 Bug 修复
        ├── 安全漏洞补丁
        └── 准备下一个项目的 Bring-up
```

> 💡 **真实感受**：Bring-up 阶段是最有趣的——从一块"砖"变成一个能跑系统的设备，那种成就感无可替代。但到了维护阶段，修各种 corner case 的 Bug 就比较考验耐心了。

## 六、这个职业的"爽"和"不爽"

### 爽的地方

1. **离硬件近**：能看到自己的代码直接控制硬件，LED 亮了、屏幕亮了、系统起来了——这是纯软件开发感受不到的
2. **门槛高 = 竞争少**：不是谁都能干固件，懂硬件又懂软件的人相对少
3. **稳定性好**：固件不像互联网业务那么卷，不会"996 赶版本上线"
4. **全球化**：UEFI/ACPI 是全球标准，技术积累在哪里都能用

### 不爽的地方

1. **调试难**：没有断点（很多时候只能靠串口打 log），Bug 可能需要 1000 次重启才能复现
2. **依赖多**：固件的 Bug 可能是硬件问题，也可能是芯片设计问题，不一定能靠改代码修
3. **文档少**：很多芯片手册写得含糊不清（"reserved"满天飞），得靠猜 + 实验
4. **薪资天花板**：和互联网大厂比，芯片/固件公司的薪资通常低一些（但正在快速改善）

## 七、本篇小结

```
固件工程师日常 = 看 Bug 列表 + 接串口 + 翻代码 + 查手册 + 改代码 + 编译 + 刷机 + 跑测试 + 循环

核心能力 = C 语言 + 硬件寄存器 + 串口调试 + 看芯片手册 + 耐心

ARM BSP vs x86 BIOS = 平台不同，方法论相同

项目节奏 = Bring-up（最爽）→ 驱动集成（最忙）→ 优化联调 → 量产维护
```
