# 什么是 UEFI 驱动模型？和操作系统驱动有什么区别？

## 写在前面

上一篇我们聊了 UEFI 的模块化思想——几百个独立模块通过 Protocol 通信。

那你可能会问：**UEFI 驱动和 Windows/Linux 的驱动有什么区别？**

毕竟都叫"驱动"，都是用来和硬件打交道的。但它们的运行环境、生命周期、设计目标完全不同。

搞清楚这个区别，你就能理解为什么固件世界需要一套自己的驱动模型，以及为什么操作系统启动后固件驱动会被"踢掉"。

---

## 一、先搞清楚：驱动是干啥的？

不管在什么环境里，驱动的本质都一样——**在软件和硬件之间架一座桥**。

```
应用程序
    ↓ 调用标准接口
驱动程序（翻译官）
    ↓ 操作硬件寄存器
硬件设备
```

你的程序说"给我读第 100 号扇区的数据"，驱动负责把这句话翻译成硬件能理解的寄存器操作序列，然后把数据搬回来。

UEFI 驱动和 OS 驱动都干这个活，但它们的"工作合同"完全不同。

---

## 二、UEFI 驱动 vs OS 驱动：核心区别

| 维度 | UEFI 驱动 | OS 驱动（Windows/Linux） |
|------|----------|----------------------|
| **运行时机** | 操作系统启动**之前** | 操作系统启动**之后** |
| **运行环境** | 无 OS，直接跑在裸机上 | 在内核态（Ring 0）运行 |
| **生命周期** | 从固件加载到 OS 接管（然后被替换） | 随 OS 运行，可热插拔 |
| **主要目的** | 让硬件能初始化、让 OS 能被加载 | 让硬件在 OS 中完整功能可用 |
| **驱动数量** | 10~50 个（只覆盖启动必需设备） | 几千个（覆盖所有设备） |
| **功能深度** | 基本功能（能用就行） | 完整功能（性能优化、高级特性） |
| **编程语言** | C（EDK2 框架） | C/C++/Rust（内核框架） |
| **接口规范** | UEFI Protocol | WDM/WDF（Windows）/ Linux 内核 API |

### 一个形象的比喻

UEFI 驱动就像**急救医生**——病人（硬件）送来后，先做基本检查、打上生理盐水、维持生命体征。目标是让病人稳定下来，等专科医生（OS 驱动）来接手。

OS 驱动是**专科医生**——做全面检查、制定治疗方案、给最优治疗。

急救医生不需要也不会做手术（UEFI 驱动不需要发挥硬件的全部性能），但没有急救医生的第一步处理，专科医生根本没机会出场。

## 三、UEFI 驱动的生命周期

一个 UEFI 驱动从出生到退休，经历这些阶段：

```
固件启动
    ↓
DXE Dispatcher 加载驱动的 .efi 文件到内存
    ↓
调用驱动的 Entry Point（入口函数）
    ↓
驱动注册 Driver Binding Protocol
（告诉系统："我能管理 NVMe 设备"）
    ↓
系统发现新设备时，调用 Supported() 问驱动：
    "这个设备你能管吗？"
    ↓
驱动回答 "能！" → 系统调用 Start()
    驱动初始化设备，安装设备 Protocol
    ↓
其他模块通过 Protocol 使用设备
（比如文件系统驱动通过 Block IO 读磁盘）
    ↓
OS 启动，调用 ExitBootServices()
    ↓
大部分 UEFI 驱动被"退休"，OS 加载自己的驱动接管设备
```

> 💡 **关键理解**：UEFI 驱动的使命在 OS 启动那一刻就结束了。`ExitBootServices()` 就是那个"交接仪式"——固件把控制权交给操作系统，大部分 Boot Services 驱动被清理出内存。

只有 Runtime 驱动（如变量服务 Variable DXE）会继续存活，因为 OS 启动后还需要通过它读写 UEFI 变量。

---

## 四、UEFI 驱动的三大分类

### 4.1 Boot Service 驱动（最多的一类）

- 在启动阶段工作，OS 接管后被卸载
- 大部分驱动属于这类：PCI、USB、NVMe、网络、文件系统……

### 4.2 Runtime 驱动

- OS 启动后仍然存活
- 例子：Variable DXE（UEFI 变量服务）、RTC（实时时钟）
- OS 通过 `RuntimeServicesTable` 调用它们

### 4.3 SMM 驱动

- 运行在系统管理模式（SMM）中，和其他所有代码隔离
- 处理电源按键、BIOS 写保护等安全敏感操作
- OS 看不到它们，但它们一直在后台运行

```
┌─────────────────────────────┐
│        UEFI 驱动分类         │
├──────────┬──────────┬───────┤
│Boot Svc  │ Runtime  │  SMM  │
│驱动      │ 驱动     │ 驱动  │
│          │          │       │
│OS 启动后 │OS 启动后 │始终在后│
│被卸载    │继续存活  │台运行  │
│          │          │       │
│PCI/USB/  │Variable/ │电源管理│
│NVMe/Net  │RTC       │安全    │
└──────────┴──────────┴───────┘
```

## 五、UEFI Driver Binding Protocol

UEFI 驱动模型的核心机制是 **Driver Binding Protocol**，它包含三个函数：

```c
typedef struct {
    EFI_DRIVER_BINDING_SUPPORTED   Supported;  // "这设备你能管吗？"
    EFI_DRIVER_BINDING_START       Start;      // "能的话，开始干活"
    EFI_DRIVER_BINDING_STOP        Stop;       // "别干了，撤退"
} EFI_DRIVER_BINDING_PROTOCOL;
```

### 工作流程

```
系统发现一个新 PCI 设备
    ↓
遍历所有已加载的驱动，逐个调用 Supported()
    ↓
USB 驱动.Supported() → "不是我的设备，返回 NOT_FOUND"
NVMe 驱动.Supported() → "是 NVMe 控制器！返回 SUCCESS"
    ↓
系统调用 NVMe 驱动.Start()
    ↓
NVMe 驱动初始化控制器，安装 Block IO Protocol
    ↓
文件系统驱动看到 Block IO，接着 Supported() → Start()
    ↓
FAT 文件系统 Protocol 安装完成
    ↓
用户就能在 Shell 里 `ls` 看到文件了
```

> 这是一个 **"发布-订阅"** 式的设计。驱动不需要主动去找设备，系统会在设备出现时自动"撮合"驱动和设备。
>
> 对比 Linux 内核的 `probe()` 函数，思路其实是一样的。

## 六、为什么 OS 不直接用 UEFI 驱动？

既然 UEFI 驱动已经能让硬件工作了，为什么 OS 还要自己再写一套驱动？

| 原因 | 说明 |
|------|------|
| **功能不完整** | UEFI 的 NVMe 驱动只实现基本读写，没有 IO 调度、NCQ、多队列优化 |
| **性能不够** | UEFI 驱动是"能用就行"级别，OS 驱动要追求极致性能 |
| **安全模型不同** | OS 有内核态/用户态隔离，UEFI 没有 |
| **中断模型不同** | UEFI 用轮询（polling），OS 用中断驱动（interrupt-driven） |
| **内存模型不同** | UEFI 用物理地址，OS 用虚拟地址 |

> 简单说：UEFI 驱动是"启动级"驱动，OS 驱动是"生产级"驱动。就像 Demo 和正式产品的区别。

不过有一个例外——**GOP 驱动**。在某些情况下（比如 OS 还没加载到显卡驱动），OS 会暂时使用 UEFI GOP 驱动提供的基本显示功能。这就是为什么 Windows 安装过程中能有图形界面但分辨率很低。

---

## 七、总结

| 维度 | UEFI 驱动 | OS 驱动 |
|------|----------|---------|
| 使命 | 初始化硬件，让 OS 能启动 | 让硬件发挥全部功能 |
| 生存期 | 固件阶段（大部分在 ExitBootServices 后消亡） | OS 运行全程 |
| 性能要求 | 能用就行 | 极致优化 |
| 核心接口 | Driver Binding Protocol | WDM/WDF / Linux 内核 API |
| 三大函数 | Supported() / Start() / Stop() | probe() / remove()（Linux） |
