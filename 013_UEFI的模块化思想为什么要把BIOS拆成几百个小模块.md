# UEFI 的模块化思想：为什么要把 BIOS 拆成几百个小模块？

## 写在前面

如果你翻开一个 EDK2 工程的编译日志，你会看到几百行 `Loading driver xxx`。

几百个模块？写一个固件要这么多东西？

传统 BIOS 不是一坨代码就跑起来了吗？为什么 UEFI 要搞这么复杂？

这个问题的答案，藏在软件工程最古老也最深刻的一个原则里——**分治（Divide and Conquer）**。

UEFI 把固件从"一锅炖"变成了"乐高积木"，每块积木各司其职，想用哪块拼哪块。

今天我们来聊聊为什么要这么干，以及这么干带来了什么好处。

---

## 一、传统 BIOS：一坨面条代码

传统 BIOS 的代码结构是这样的：

```
bios_main() {
    初始化 CPU();
    初始化内存();
    初始化 PCI();
    初始化 USB();
    初始化显卡();
    初始化键盘();
    检测硬盘();
    加载 MBR();
    跳转到操作系统();
}
```

看着很清晰？那是因为只有 6 行伪代码。

真实的 BIOS 代码里，这些函数之间互相调用、互相依赖、共享全局变量，一个函数改了某个寄存器状态，另一个函数就崩了。30 年积累下来，代码变成了一碗纠缠不清的意大利面。

> 这就是程序员闻之色变的 **"屎山代码"（Spaghetti Code）**。

问题有多严重？

| 问题 | 后果 |
|------|------|
| **紧耦合** | 改 USB 驱动可能导致 PCI 枚举出问题 |
| **不可复用** | 换一个平台，大部分代码要重写 |
| **不可测试** | 没有独立模块，只能烧进去跑黑盒 |
| **不可裁剪** | 不需要网络功能？抱歉，拆不掉 |
| **多人协作难** | Intel 改完代码，OEM 的定制全炸 |

---

## 二、UEFI 的解法：模块化

UEFI/EDK2 的核心思想：**把固件拆成几百个独立的模块，每个模块只做一件事，模块之间通过标准接口通信。**

```
传统 BIOS                      UEFI/EDK2
┌──────────────────┐          ┌──────┐ ┌──────┐ ┌──────┐
│                  │          │PCI 驱动│ │USB 驱动│ │NVMe  │
│  一坨代码        │    →     └───┬──┘ └───┬──┘ └───┬──┘
│  (紧耦合)        │              │        │        │
│                  │          ┌───┴────────┴────────┴───┐
└──────────────────┘          │   标准接口 (Protocol)     │
                              └────────────────────────┘
```

### EDK2 中的模块分类

| 模块类型 | 缩写 | 运行阶段 | 举例 |
|---------|------|---------|------|
| SEC 模块 | SEC | 最早期 | 安全验证、CAR 初始化 |
| PEI 模块 | PEIM | 内存初始化前 | 内存初始化（MRC） |
| DXE 驱动 | DXE_DRIVER | 主要阶段 | PCI 总线驱动、USB 驱动 |
| UEFI 驱动 | UEFI_DRIVER | DXE 阶段 | NVMe 驱动、网络驱动 |
| UEFI 应用 | UEFI_APPLICATION | BDS 阶段 | UEFI Shell、启动管理器 |
| SMM 驱动 | DXE_SMM_DRIVER | SMM 模式 | 电源管理、安全相关 |
| Runtime 驱动 | DXE_RUNTIME_DRIVER | 跨越到 OS | 变量服务、RTC 服务 |

一个典型的 UEFI 固件大概包含 **200~500 个模块**。

## 三、模块化的好处

### 3.1 解耦：改一个不影响另一个

每个模块独立编译、独立加载。你修改 NVMe 驱动的代码，PCI 驱动完全不受影响。

```
修改 NvmExpressDxe.efi
    ↓
只重新编译这一个模块
    ↓
替换到固件镜像中
    ↓
其他 199 个模块完全不变
```

### 3.2 可复用：一次开发，多平台使用

EDK2 的通用驱动（PCI、USB、NVMe、网络栈）可以在不同平台上复用：

```
NvmExpressDxe.efi
    ├── Intel 平台 ✓
    ├── AMD 平台 ✓
    ├── ARM 平台 ✓
    └── QEMU 虚拟机 ✓
```

因为这些驱动只依赖 UEFI 标准接口（Protocol），不依赖具体硬件。硬件差异被底层驱动屏蔽了。

### 3.3 可裁剪：不需要的功能直接删掉

嵌入式设备不需要网络功能？从 DSC 文件中删掉网络相关模块就行：

```ini
# 删掉这些行，固件里就没有网络栈了
# NetworkPkg/TcpDxe/TcpDxe.inf
# NetworkPkg/UdpDxe/UdpDxe.inf
# NetworkPkg/Dhcp4Dxe/Dhcp4Dxe.inf
```

传统 BIOS 想删功能？祝你好运，搞不好删完就开不了机了。

### 3.4 可测试：单个模块独立验证

EDK2 支持模块级的单元测试。你可以只编译和测试一个驱动，不需要启动整个固件：

```bash
build -m SecurityPkg/Library/SomeLib/SomeLib.inf
# 只编译这一个库
```

### 3.5 多团队协作：各管各的

在实际的 BSP 开发中，不同团队负责不同模块：

```
Intel 硅平台团队 → 内存初始化、PCH 驱动、FSP
BIOS 框架团队    → DXE 核心、Boot Manager、HII
驱动团队         → USB、NVMe、网络驱动
安全团队         → Secure Boot、TPM 驱动
OEM 定制团队     → Logo、Setup 菜单、特殊功能
```

大家各改各的模块，通过 Protocol 接口约定通信方式，互不干扰。

## 四、模块之间怎么通信？

既然模块是独立的，那 PCI 驱动怎么告诉 NVMe 驱动："嘿，我发现了一个 NVMe 控制器"？

答案就是 UEFI 最核心的设计：**Protocol（协议）**。

Protocol 本质上是一张**函数指针表**（结构体里装着一堆函数指针），挂在 Handle（句柄）上。

```c
// 一个简化的 Block IO Protocol
typedef struct {
    EFI_BLOCK_READ     ReadBlocks;   // 读磁盘
    EFI_BLOCK_WRITE    WriteBlocks;  // 写磁盘
    EFI_BLOCK_RESET    Reset;        // 重置设备
    UINT32             BlockSize;    // 块大小
} EFI_BLOCK_IO_PROTOCOL;
```

NVMe 驱动实现了 `ReadBlocks` 和 `WriteBlocks`，然后把这个 Protocol 注册到系统中。

文件系统驱动不需要知道底层是 NVMe 还是 SATA，它只需要找到 `EFI_BLOCK_IO_PROTOCOL`，调用 `ReadBlocks()` 就能读数据。

```
文件系统驱动 → 调用 Block IO Protocol → NVMe 驱动 → 读取 NVMe 硬件
                                   或 → SATA 驱动 → 读取 SATA 硬件
                                   或 → USB 存储驱动 → 读取 U 盘
```

> 这就是**面向接口编程**——和 Java 的 Interface、Go 的 Interface、Rust 的 Trait 思想完全一样，只不过 UEFI 用 C 语言 + 函数指针表实现的。

## 五、用一个比喻总结

传统 BIOS 像一家**夫妻小店**：
- 老板一个人采购、做饭、端盘子、收银、打扫卫生
- 效率？就那样。想扩大规模？几乎不可能
- 老板生病了？整个店关门

UEFI 像一家**连锁餐厅**：
- 采购部门、厨房、前台、收银、保洁，各部门独立运作
- 想加一个新菜品？只需要厨房增加一个厨师
- 不同门店（平台）可以复用同一套管理体系
- 某个部门出问题？替换那个部门就行，不影响其他部门

---

## 六、总结

| 维度 | 传统 BIOS | UEFI/EDK2 |
|------|----------|-----------|
| 代码结构 | 单一大文件 | 几百个独立模块 |
| 模块间通信 | 全局变量、直接调用 | Protocol（标准接口） |
| 可维护性 | 改一处崩一片 | 模块独立，互不影响 |
| 可复用性 | 换平台重写 | 通用模块跨平台复用 |
| 可裁剪性 | 删不掉 | DSC 文件删一行就行 |
| 多人协作 | 冲突不断 | 各管各的模块 |
