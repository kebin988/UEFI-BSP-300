## 写在前面

SEC 阶段结束时，我们有了 CAR（几百 KB 的临时 RAM）和一个可以跑 C 代码的栈。但离"正常运行"还差得远——几百 KB 连跑一个现代文本编辑器都不够。

PEI 阶段的使命就是：**在资源极其有限的条件下，把系统从"勉强能跑"推进到"内存充足，万事俱备"。**

它是整个 UEFI 启动中最**拧巴**的阶段——既要完成复杂的硬件初始化，又只有几百 KB 内存可用。就像让你在一间 5 平米的房间里组装一台汽车。

## 一、PEI 的全景视角

```
SEC 交出控制权
    │
    │  携带：CAR 信息、Boot FV 地址
    ▼
┌──────────────────────────────────────────────────────────┐
│  PEI Core 启动                                            │
│                                                          │
│  1. 初始化 PEI Services（内存分配、PPI 数据库）          │
│  2. 创建 HOB List（为 DXE 准备交接数据）                 │
│  3. 调度并执行 PEIM 模块                                  │
│     ├── Platform Init PEIM（芯片组基础配置）              │
│     ├── CPU Init PEIM（微码加载、CPU 配置）               │
│     ├── Memory Init PEIM（★ DRAM 初始化 ★）             │
│     ├── Silicon Init PEIM（PCH/SoC 配置）                │
│     ├── S3 Resume PEIM（休眠唤醒）                        │
│     └── Recovery PEIM（固件恢复）                          │
│                                                          │
│  ═══════════════ DRAM 初始化完成 ═══════════════          │
│                                                          │
│  4. 把 CAR 数据迁移到 DRAM                               │
│  5. 继续调度后续 PEIM                                     │
│  6. 找到 DXE Core，跳转过去                              │
│                                                          │
│  传递给 DXE：HOB List（内存布局、平台信息）              │
└──────────────────────────────────────────────────────────┘
    │
    ▼
DXE Core 接管
```

## 二、PEI 的"两个世界"

PEI 阶段以 DRAM 初始化为分界线，分成两个截然不同的世界：

| | 前半段（Pre-Memory） | 后半段（Post-Memory） |
|--|---------------------|---------------------|
| 内存来源 | CAR（几百KB） | DRAM（几GB） |
| 代码执行 | XIP（从 Flash 直接执行） | 可以复制到 RAM 执行 |
| 模块数量 | 少（5-10个） | 多（10-20+） |
| 能做的事 | 基础硬件初始化 | 更复杂的初始化 |
| 速度 | 慢（Flash 读取） | 快（RAM 执行） |

```
时间线：

SEC ──→ PEI Pre-Memory ──→ [DRAM OK!] ──→ PEI Post-Memory ──→ DXE
         │                                  │
         │ 只有 CAR                          │ 有完整 DRAM
         │ 从 Flash XIP                      │ 从 RAM 执行
         │ 最精简的 PEIM                     │ 更多 PEIM 可以跑
```

## 三、PEI Core 的内部结构

PEI Core 是 PEI 阶段的"操作系统"，它提供 **PEI Services** 表：

```c
typedef struct {
    EFI_TABLE_HEADER          Hdr;
    
    // 安装和查找 PPI
    EFI_PEI_INSTALL_PPI       InstallPpi;
    EFI_PEI_REINSTALL_PPI     ReInstallPpi;
    EFI_PEI_LOCATE_PPI        LocatePpi;
    EFI_PEI_NOTIFY_PPI        NotifyPpi;
    
    // 内存管理
    EFI_PEI_GET_BOOT_MODE     GetBootMode;
    EFI_PEI_SET_BOOT_MODE     SetBootMode;
    EFI_PEI_ALLOCATE_PAGES    AllocatePages;
    EFI_PEI_ALLOCATE_POOL     AllocatePool;     // 从 CAR 堆分配
    EFI_PEI_FREE_PAGES        FreePages;
    
    // HOB 操作
    EFI_PEI_GET_HOB_LIST      GetHobList;
    EFI_PEI_CREATE_HOB        CreateHob;
    
    // 固件卷操作
    EFI_PEI_FFS_FIND_NEXT_VOLUME  FfsFindNextVolume;
    EFI_PEI_FFS_FIND_NEXT_FILE    FfsFindNextFile;
    EFI_PEI_FFS_FIND_SECTION      FfsFindSectionData;
    
    // 其他
    EFI_PEI_INSTALL_PEI_MEMORY    InstallPeiMemory;  // 通知"内存好了"
    EFI_PEI_RESET_SYSTEM          ResetSystem;
} EFI_PEI_SERVICES;
```

和 DXE 的 Boot Services 比起来，PEI Services 简陋得多——因为资源有限，只提供最基本的功能。

## 四、PEI 的内存管理

### 4.1 Pre-Memory 阶段

在 DRAM 初始化之前，所有内存分配都来自 **CAR 堆**：

```c
// 从 CAR 堆分配 4KB
VOID *Buffer;
Status = PeiServices->AllocatePages(
    PeiServices,
    EfiBootServicesData,
    1,                    // 1 page = 4KB
    &PhysicalAddress
);

// 注意：CAR 堆只有几百 KB，分配可能失败！
// 而且没有 Free —— 分配了就不能释放
```

### 4.2 InstallPeiMemory：重要的转折点

当 Memory Init PEIM 成功初始化 DRAM 后，它调用：

```c
// "报告！内存初始化完成了！"
Status = PeiServices->InstallPeiMemory(
    PeiServices,
    MemoryBegin,      // DRAM 可用区域起始
    MemoryLength      // DRAM 可用区域大小
);
```

这个调用是 PEI 阶段的**分水岭**：
- 之前：只有 CAR
- 之后：有了 DRAM，PEI Core 会把数据迁移到 DRAM，后续分配从 DRAM 进行

### 4.3 Post-Memory 阶段

DRAM 可用后：
- AllocatePages 从 DRAM 分配
- 新加载的 PEIM 可以被复制到 RAM 执行（不再 XIP）
- HOB List 有充足空间

## 五、XIP (Execute In Place) ——从 Flash 直接执行

Pre-Memory 阶段的代码直接从 **Flash** 执行，不复制到 RAM：

```
正常执行（有 RAM 后）：
  Flash 中的代码 → 复制到 RAM → CPU 从 RAM 执行（快）

XIP 执行（没有 RAM 时）：
  Flash 中的代码 → CPU 直接从 Flash 执行（慢但可行）
```

XIP 的特点：
- ✅ 不需要 RAM 就能执行
- ❌ 很慢（SPI Flash 读速度约 50 MB/s，DRAM 约 50 GB/s，差 1000 倍）
- ❌ 只能读不能写（Flash 是只读的执行环境）
- ❌ 不能修改全局变量（全局变量在 .data 段，属于 Flash 中的只读区域）

```c
// 在 XIP 模式下：
static UINT32 gCounter = 0;  // 这个全局变量在 Flash 中！

void Increment(void) {
    gCounter++;  // ❌ 写 Flash = 失败或无效！
    // Pre-Memory PEIM 不能使用可写全局变量
}

// 正确做法：把状态存在 CAR 的堆分配中
void CorrectIncrement(EFI_PEI_SERVICES **PeiServices) {
    MY_DATA *Data;
    PeiServices->AllocatePool(PeiServices, sizeof(MY_DATA), &Data);
    Data->Counter++;  // ✅ 这是在 CAR 中，可读写
}
```

## 六、PEI 阶段的 Boot Mode

PEI 需要知道当前是什么类型的启动，以决定执行哪些 PEIM：

```c
typedef enum {
    BOOT_WITH_FULL_CONFIGURATION,          // 正常冷启动
    BOOT_WITH_MINIMAL_CONFIGURATION,       // 快速启动
    BOOT_ASSUMING_NO_CONFIGURATION_CHANGES,// 无变更启动
    BOOT_WITH_FULL_CONFIGURATION_PLUS_DIAGNOSTICS, // 诊断启动
    BOOT_WITH_DEFAULT_SETTINGS,            // 恢复默认设置
    BOOT_ON_S4_RESUME,                     // S4 休眠恢复
    BOOT_ON_S5_RESUME,                     // S5 重启
    BOOT_ON_S2_RESUME,                     // S2
    BOOT_ON_S3_RESUME,                     // S3 睡眠唤醒（★重要★）
    BOOT_ON_FLASH_UPDATE,                  // 固件更新
    BOOT_IN_RECOVERY_MODE,                 // 恢复模式
} EFI_BOOT_MODE;
```

最关键的是 **S3 Resume**（睡眠唤醒）：
- 普通启动：跑所有 PEIM → DXE → BDS → 加载 OS
- S3 唤醒：跑少量 PEIM → 恢复内存训练 → 直接跳到 OS 恢复点

```
S3 Resume 的快速路径：

PEI ─→ Memory Init (从保存的参数快速恢复) ─→ 跳过 DXE/BDS ─→ 直接到 OS
        约 200ms                                               总计 < 1s

vs 正常启动：
PEI ─→ Memory Init ─→ DXE (加载所有驱动) ─→ BDS ─→ OS
        约 200ms       约 3-5s                      总计 5-10s
```

## 七、PEI 的调度器（Dispatcher）

PEI Core 内置调度器，按依赖顺序执行 PEIM：

```
PEI Dispatcher 循环：

1. 扫描 Boot FV 中的所有 PEIM 文件
2. 检查每个 PEIM 的 Dependency Expression（DEPEX）
3. 如果依赖满足 → 执行该 PEIM
4. PEIM 可能安装新 PPI → 触发其他 PEIM 的依赖变为满足
5. 回到步骤 2，直到没有新 PEIM 可执行

伪代码：
do {
    DispatchedThisRound = FALSE;
    for each PEIM in FirmwareVolume {
        if (PEIM not yet dispatched && DEPEX satisfied) {
            Execute(PEIM);
            DispatchedThisRound = TRUE;
        }
    }
} while (DispatchedThisRound);
```

## 八、PEI 结束：跳转到 DXE

所有 PEIM 执行完毕后，PEI Core 执行最后一步——找到并启动 DXE Core：

```c
// PEI 的最后一步
VOID PeiCoreEnd(EFI_PEI_SERVICES **PeiServices)
{
    EFI_DXE_IPL_PPI  *DxeIpl;  // DXE Initial Program Loader
    EFI_HOB_LIST     *HobList;
    
    // 1. 找到 DXE IPL PPI
    PeiServices->LocatePpi(
        PeiServices, &gEfiDxeIplPpiGuid, 0, NULL, &DxeIpl
    );
    
    // 2. 获取 HOB List
    PeiServices->GetHobList(PeiServices, &HobList);
    
    // 3. 调用 DXE IPL，启动 DXE Core
    //    （内部会加载 DXE Core 到 RAM，然后跳转过去）
    DxeIpl->Entry(DxeIpl, PeiServices, HobList);
    
    // 永远不会到这里
}
```

DXE IPL (Initial Program Loader) 做的事：
1. 从 Firmware Volume 中找到 DXE Core 文件
2. 把 DXE Core 加载到 DRAM
3. 传递 HOB List
4. 跳转到 DXE Core 入口

## 九、PEI 阶段的常见问题

| 问题 | 原因 | 解决方法 |
|------|------|---------|
| 内存分配失败 | CAR 堆用完了 | 减少 Pre-Memory 阶段的分配 |
| PEIM 没被调度 | DEPEX 依赖不满足 | 检查依赖的 PPI 是否被安装 |
| 全局变量写入失败 | XIP 模式下不可写 | 用 AllocatePool 代替 |
| S3 唤醒慢 | Memory Training 重跑 | 保存/恢复训练参数 |
| 启动挂死在 PEI | PEIM 无限循环或崩溃 | 用 Post Code 定位 |

## 十、总结

```
PEI 的核心矛盾：

  任务重大：初始化 CPU、内存、芯片组
  资源匮乏：只有几百 KB 的 CAR

  解决方案：
  ├── XIP 执行（不占 RAM）
  ├── 精简的 PEI Services（最小化基础设施）
  ├── 分阶段执行（Pre-Memory vs Post-Memory）
  └── 快速路径（S3 跳过大部分初始化）
```

| PEI 关键概念 | 说明 |
|-------------|------|
| PEI Core | PEI 的"微型 OS" |
| PEIM | PEI 模块（类似 DXE 的驱动） |
| PPI | PEI 阶段的通信接口 |
| HOB | 传递给 DXE 的信息包 |
| XIP | 从 Flash 直接执行代码 |
| Pre/Post Memory | 以 DRAM 初始化为分界 |
| Boot Mode | 决定走哪条启动路径 |
| DXE IPL | PEI 到 DXE 的跳板 |
