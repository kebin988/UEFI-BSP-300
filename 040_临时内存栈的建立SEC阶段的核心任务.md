## 写在前面

上一篇讲了 CAR 的原理——用 Cache 模拟 RAM。但 CAR 建好之后，**栈怎么设置**？设多大？多核 CPU 每个核心的栈怎么安排？PEI Core 怎么接管？

这些问题就是今天的主角。栈的建立看似简单（一条 `mov esp, xxx` 指令），但背后有很多工程决策。

---

## 一、为什么栈这么重要？

没有栈，C 语言的以下功能全部无法使用：

```c
// 函数调用 → 需要栈保存返回地址
void FuncA(void) { FuncB(); }

// 局部变量 → 需要栈空间
void FuncC(void) { int x = 42; }

// 参数传递 → 需要栈（或寄存器+栈）
void FuncD(int a, int b, int c) { ... }

// 递归 → 需要多层栈帧
int Factorial(int n) { return n * Factorial(n-1); }
```

汇编代码只要有寄存器就能跑，但 C 代码**必须有栈**。SEC 阶段从汇编切到 C 的那个瞬间，栈必须已经设好。

---

## 二、栈的建立时机

在整个 SEC 启动流程中，栈建立的精确位置：

```
CPU Reset
  │
  ▼
Reset Vector: JMP to SEC Entry (汇编)
  │
  ▼
切换到保护模式/长模式 (汇编)          ← 此时没有栈
  │
  ▼
配置 CAR (MTRR + NEM) (汇编)         ← 此时没有栈
  │
  ▼
══════════════════════════════════════
  设置 ESP/RSP → CAR 区域             ← ★ 栈建立！
══════════════════════════════════════
  │
  ▼
CALL SecStartup (C 函数)             ← 从这里开始可以跑 C 了
  │
  ▼
SEC 完成 → 跳转到 PEI Core
```

**精确时间点**：CAR 配置完成后、第一次 CALL 指令之前。

## 三、栈的位置和大小选择

### 3.1 栈放在 CAR 区域的哪里？

通常放在 CAR 区域的**顶部**（高地址端），因为 x86 栈向下增长：

```
CAR_BASE + CAR_SIZE
┌─────────────────────────┐  ← ESP 初始值（栈顶）
│                         │
│         栈              │  ↓ 向下增长
│    (Stack grows down)   │
│                         │
├─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─┤  ← 栈底（用完就爆了）
│                         │
│     数据/堆区域         │  ← PEI 的 HOB、PPI 等
│                         │
└─────────────────────────┘
CAR_BASE
```

### 3.2 栈给多大？

这是一个**平台工程决策**，取决于：

| 因素 | 影响 |
|------|------|
| CPU Cache 总大小 | CAR 最大容量的上限 |
| PEI 阶段需要的堆空间 | 和栈竞争同一块 CAR |
| SEC/PEI 代码的栈使用量 | 决定最小栈大小 |
| 是否有嵌套调用/递归 | 深调用栈需要更多空间 |

典型配置：

| 平台 | CAR 总大小 | 栈大小 | 堆大小 |
|------|-----------|--------|--------|
| Intel Atom | 256 KB | 64 KB | 192 KB |
| Intel Core (Client) | 512 KB | 128 KB | 384 KB |
| Intel Xeon (Server) | 1 MB | 256 KB | 768 KB |
| AMD Zen | 512 KB | 128 KB | 384 KB |

### 3.3 EDK2 中的配置

在 EDK2 的平台 DSC 文件中：

```ini
# Platform.dsc 中的 PCD 配置
[PcdsFixedAtBuild]
  # CAR 区域
  gUefiCpuPkgTokenSpaceGuid.PcdTemporaryRamBase|0xFEF00000
  gUefiCpuPkgTokenSpaceGuid.PcdTemporaryRamSize|0x00080000  # 512KB

  # 其中栈占多少
  gEfiMdePkgTokenSpaceGuid.PcdPeiTemporaryRamStackSize|0x20000  # 128KB
```

代码中设置栈：

```nasm
; EDK2 SecEntry.nasm (x64 示例)
;
; CAR 已经配置好了
; 设置栈指针

mov     rsp, FixedPcdGet32 (PcdTemporaryRamBase)
add     rsp, FixedPcdGet32 (PcdTemporaryRamSize)
; RSP 现在指向 CAR 区域顶部

; 调用 C 入口
; RCX = TemporaryRamBase
; RDX = TemporaryRamSize
mov     rcx, FixedPcdGet32 (PcdTemporaryRamBase)
mov     rdx, FixedPcdGet32 (PcdTemporaryRamSize)
call    SecStartup
```

## 四、栈的划分：SEC 和 PEI 共享

SEC 建立栈后，会把整个临时 RAM 的信息传递给 PEI Core。PEI Core 会重新划分这块空间：

```c
// SEC 传给 PEI 的信息
typedef struct {
    UINTN                   DataSize;
    VOID                    *BootFirmwareVolumeBase;
    UINTN                   BootFirmwareVolumeSize;
    VOID                    *TemporaryRamBase;      // CAR 起始
    UINTN                   TemporaryRamSize;       // CAR 大小
    VOID                    *PeiTemporaryRamBase;   // PEI 堆起始
    UINTN                   PeiTemporaryRamSize;    // PEI 堆大小
    VOID                    *StackBase;             // 栈底
    UINTN                   StackSize;              // 栈大小
} EFI_SEC_PEI_HAND_OFF;
```

PEI Core 收到后的内存视图：

```
                    TemporaryRamBase + TemporaryRamSize
                    ┌─────────────────────────┐
                    │                         │
                    │    Stack                 │  StackSize
                    │    (SEC 和 PEI 共用)     │
                    │                         │
StackBase ─────────├─────────────────────────┤
                    │                         │
                    │    PEI Heap             │  PeiTemporaryRamSize
                    │    (HOB, PPI, PEIM数据) │
                    │                         │
PeiTemporaryRamBase├─────────────────────────┤
                    │    (可能的 SEC 数据)     │
                    └─────────────────────────┘
                    TemporaryRamBase
```

## 五、多核环境下的栈安排

现代 CPU 都是多核的。上电时只有一个核在跑代码（**BSP** - Bootstrap Processor），其他核处于等待状态（**AP** - Application Processor）。

### 5.1 BSP 的栈

BSP 是第一个执行的核心，它拥有主栈：

```
BSP 使用整个 CAR 栈区域
SEC → PEI 都在 BSP 上单线程执行
```

### 5.2 AP 什么时候醒来？

通常在 PEI 后期或 DXE 阶段，BSP 会唤醒 AP：

```c
// BSP 发送 INIT-SIPI-SIPI 唤醒 AP
SendInitSipiSipi(ApStartupVector);
```

### 5.3 AP 的栈从哪来？

AP 醒来后也需要栈！方案：

```
方案 A：在 DRAM 中分配（如果 DRAM 已初始化）
  - DXE 阶段唤醒 AP → DRAM 可用 → 每个 AP 分配独立栈

方案 B：在 CAR 中预留（如果 PEI 阶段就需要多核）
  - CAR 划分多个栈区域 → 一个给 BSP，其余给 AP
  - 每个 AP 栈很小（8-16 KB）

实际常见做法：
┌─────────────────────────┐
│ BSP Stack (128KB)       │  ← BSP 独占大栈
├─────────────────────────┤
│ AP#1 Stack (16KB)       │  ← 唤醒时分配
│ AP#2 Stack (16KB)       │
│ AP#3 Stack (16KB)       │
│ ...                     │
├─────────────────────────┤
│ PEI Heap               │
└─────────────────────────┘
```

## 六、栈溢出：最可怕的早期启动 Bug

### 6.1 为什么栈溢出特别危险

在操作系统中，栈溢出会触发 Page Fault（保护页机制）。但在 CAR 阶段：
- **没有页表**（可能还没配）
- **没有异常处理**（IDT 可能还没设好）
- **没有任何保护**

栈溢出 = 写到了 CAR 区域之外 = **静默数据损坏** 或 **覆盖其他区域的数据**

```
正常情况：
  ┌─────── Stack ───────┐
  │ Frame A             │
  │ Frame B             │
  │ Frame C             │  ← ESP
  ├─────── 边界 ────────┤
  │ PEI Heap 数据       │  ← 安全

栈溢出：
  ┌─────── Stack ───────┐
  │ Frame A             │
  │ Frame B             │
  │ Frame C             │
  │ Frame D             │
  │ Frame E             │
  ├─ ─ ─ ─ 越界了！─ ─ ─┤
  │ ████████████████████ │  ← PEI 堆被栈覆盖！
  │ █ 数据损坏 ████████ │  ← HOB List 被破坏！
  └─────────────────────┘
```

### 6.2 常见导致栈溢出的代码

```c
// 1. 大型局部数组
EFI_STATUS BadInit(VOID) {
    CHAR16 BigString[4096];     // 8KB 在栈上！
    UINT8  Buffer[32768];       // 32KB！直接爆栈
    // ...
}

// 2. 深层递归
EFI_STATUS RecursiveSearch(VOID *Ptr, UINTN Depth) {
    UINT8 LocalData[256];
    // 递归 100 层 = 100 × 256 = 25KB 栈消耗
    return RecursiveSearch(Next, Depth + 1);
}

// 3. 调用链太深
// A() → B() → C() → D() → ... → Z()
// 每层 100-500 字节的栈帧，20 层就是 2-10KB
```

### 6.3 栈保护手段

```c
// 1. 栈哨兵（Stack Sentinel）
#define STACK_SENTINEL  0xA5A5A5A5

// SEC 启动时在栈底写入哨兵值
*(UINT32 *)(StackBase) = STACK_SENTINEL;

// 定期检查
if (*(UINT32 *)(StackBase) != STACK_SENTINEL) {
    // 栈溢出了！
    DEBUG((DEBUG_ERROR, "Stack overflow detected!\n"));
    CpuDeadLoop();
}

// 2. 编译时检查
// 某些编译器支持 -fstack-usage 输出每个函数的栈使用量
// 人工审查大栈使用者
```

## 七、从临时栈迁移到永久栈

当 DRAM 初始化完成后，需要把栈从 CAR 迁移到 DRAM：

```
迁移前：              迁移后：
┌── CAR ──┐          ┌── DRAM ──────────────┐
│  Stack   │ ──copy──→│  Stack (复制过来的)   │
│  [data]  │          │  [data]               │
├──────────┤          │                       │
│  Heap    │ ──copy──→│  Heap (复制过来的)    │
└──────────┘          │                       │
                      │  (更多空间可用)        │
                      └───────────────────────┘

ESP: 指向 CAR     →    ESP: 指向 DRAM 新位置
```

**难点**：栈里面可能有指针，指向 CAR 中的其他数据。迁移后这些指针就无效了！

```c
// 迁移前：
// ESP 在 CAR
// 栈帧中某个局部变量 ptr = 0xFEF01234 (指向 CAR 中的数据)

// 迁移后：
// ESP 在 DRAM (比如 0x7FF00000)
// 但 ptr 还是 0xFEF01234 → 指向已废弃的 CAR 区域！

// 解决方案：PEI Core 知道偏移量，会修正 HOB 中的指针
// 但栈帧里的局部指针无法修正 → 所以迁移必须在调用链的顶层进行
```

EDK2 的处理方式：

```c
// PEI Core 在栈迁移时调用 SwitchStack
// 切换到新栈后重新进入 PEI Dispatcher
// 之前栈帧中的局部变量全部失效（反正要重新跑 Dispatcher）

SwitchStack(
    (SWITCH_STACK_ENTRY_POINT)PeiCoreDispatcher,
    OldCoreData,      // 参数
    NULL,
    (VOID *)NewStackTop  // 新栈地址
);
```

## 八、不同架构的临时栈方案

| 架构 | 临时栈来源 | 说明 |
|------|-----------|------|
| **x86 (Intel/AMD)** | CAR (Cache As RAM) | Cache 模拟 RAM |
| **ARM Cortex-A** | On-chip SRAM | 片上 SRAM 天生可用 |
| **ARM Cortex-M** | 内置 Flash + SRAM | MCU 自带 |
| **RISC-V** | 平台相关 | 可能是 SRAM 或 TCM |
| **MIPS** | kseg0/kseg1 | Uncached/Cached 直接映射区 |

ARM 的例子：

```c
// ARM 平台的 SEC 入口（简化）
VOID
EFIAPI
_ModuleEntryPoint(VOID)
{
    // ARM 不需要 CAR，直接用 SRAM
    // SRAM 地址由硬件决定，比如 0x00900000
    
    // 设置栈（在 SRAM 中）
    __asm volatile ("mov sp, %0" : : "r" (SRAM_BASE + SRAM_SIZE));
    
    // 直接调用 C 代码
    CEntryPoint();
}
```

## 九、实战：OVMF 中的栈设置

OVMF（QEMU 的 UEFI 固件）是最好的学习范本，因为它不需要真正的 CAR（QEMU 模拟器中内存"天生可用"）：

```nasm
; OvmfPkg/Sec/X64/SecEntry.nasm

; OVMF 的 SEC 入口
; QEMU 中不需要 CAR，内存直接可用

global ASM_PFX(_ModuleEntryPoint)
ASM_PFX(_ModuleEntryPoint):
    ; 设置栈（OVMF 用固定地址）
    mov     rsp, SEC_TOP_OF_STACK
    
    ; 调整对齐（x64 要求 16 字节对齐）
    and     rsp, ~0xF
    
    ; 调用 C 入口
    ; RCX = BootFirmwareVolumePtr
    ; RDX = TopOfCurrentStack
    mov     rcx, [BootFvBaseAddress]
    mov     rdx, rsp
    
    sub     rsp, 0x20       ; Shadow space (x64 调用约定)
    call    ASM_PFX(SecStartup)
```

## 十、总结

```
临时栈建立的关键要素：

  何时？  → CAR 配置完成后的第一时间
  何处？  → CAR 区域的顶部（高地址端）
  多大？  → 64KB ~ 256KB（取决于平台）
  谁用？  → BSP 独占（AP 后来单独分配）
  何时结束？→ DRAM 初始化后迁移到 DRAM
```

| 关键点 | 说明 |
|--------|------|
| 栈的位置 | CAR 顶部，向下增长 |
| 大小选择 | 平台工程决策，通常 64-256KB |
| 多核处理 | BSP 有大栈，AP 各有小栈 |
| 最大风险 | 栈溢出（无保护，静默损坏） |
| 迁移难点 | 栈内指针可能失效 |
| ARM 不同 | 有 SRAM，不需要 CAR hack |

> **一句话总结**：临时栈是从汇编世界通往 C 世界的桥梁——`mov esp, xxx` 这一条指令之后，固件就真正"活"了。
